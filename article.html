一、它在做什么？—— 一个端到端的AI智能体平台

先说它是什么。

这是一套端到端的AI智能体平台（E2E AI Agent Platform）。你可以在上面创建自己的AI Agent，给它写system prompt，给它挂载能力模块（我们叫Skill），挂载知识库，开记忆开关，然后跟它对话。它在云端用 Deep Agent 引擎 + LangGraph 图编排来执行你的请求，过程中自动拆任务、调工具、写文档、出产物。

图1：Agent工作台——已创建的Agent都在这里展示

你可以理解成这是一个Agent的创建和执行平台，对标市面上那些AI Agent构建工具，但架构思路不太一样。

长远规划还有一套完整的流程设计和编排引擎，让Agent能按照预设的流程图去执行任务。加上云端+本地协同，基本上是往AI时代的流程引擎方向在走。

二、底层架构：Deep Agent + LangGraph + 百炼模型

先讲一下整体架构，因为这是我学AI Coding过程中啃得最狠的一块。

整个平台拆三层：

最核心的是运行时层，也就是 agent-runtime 模块。它是一个独立的 Python FastAPI 服务，跑在云端，底层用了两样东西：

第一个是 Deep Agent SDK。这是一套智能体的核心开发框架，提供 create_deep_agent 这样的接口。你可以理解成它是一个"Agent的脚手架"——你给它一个模型、一堆工具、一套系统提示词，它帮你组装成一个完整的Agent实例。

第二个是 LangGraph。这是LangChain团队出的图编排引擎，用来定义Agent的多步执行流程。传统的大模型调用是一问一答，但有了LangGraph，Agent可以走一个"图"：先规划，再执行，中间可以调工具，判断结果，决定下一步走哪个分支，最后汇总输出。

这两个东西搭配起来什么效果？我给你们看一段核心代码，其实挺直观的：

# Deep Agent SDK 创建Agent
agent = create_deep_agent(
    model=llm,                    # 通义千问 Plus/Max
    tools=all_tools,              # Skill里定义的工具
    system_prompt=full_prompt,    # 系统提示词
)

# LangGraph 流式执行
async for chunk in agent.astream(input_data):
    # 实时吐出内容给前端
    yield chunk

对，核心代码就这么几行。但真正难的是前面那些准备工作——怎么把用户的system prompt和Skill指令拼成完整的full_prompt，怎么把各种工具挂载上去，怎么处理对话历史，怎么做SSE流式输出。这些都在agent-runtime那一千多行代码里了。

底座模型用的是通义千问（百炼平台），通过OpenAI兼容协议调用，支持qwen-turbo/plus/max三档切换。为什么选百炼？因为便宜，而且国内访问稳定。作为个人项目来说，这个性价比很重要。

光讲底层太干了，来点实际的。你可以理解成这是一个"Agent的创建和执行平台"，对标市面上那些AI Agent构建工具（比如Dify、Coze），但架构思路不太一样。

这个设计其实挺产品经理思维的——我是一个PM的话，我怎么定义一个AI员工？先告诉他你是谁（System Prompt），再给你配工具（Skills），再给你参考资料（知识库），再决定要不要记住以前的对话（记忆开关）。这不就是 onboarding 一个新员工的逻辑吗？

3.2 对话与产物Canvas：不只是聊天

对话界面长这样：左侧是正常的对话流，右侧是一个"产物画布"（我们叫Canvas）。

图3：对话界面——与Agent实时对话，支持模型切换

为什么有Canvas？因为在真实工作场景中，你跟Agent聊天不只是为了聊天，而是要产出东西的。可能是它给你写了一段代码，可能是给你生成了一份分析报告，也可能是一个表格。这些东西不应该淹没在对话流里，得有个专门的区域来展示和操作。

所以我们的设计是：AI回复中的代码块、表格，会被自动识别为"产物卡片"。点击卡片，右侧Canvas就会展开，支持预览模式和源码模式切换，中间的拖拽条还可以左右调整宽度。你甚至可以直接把产物下载成Word文档。

这个设计是我自己比较满意的一个产品细节。它解决了一个真实痛点——"AI说了很多，但我要的东西在哪里"。

3.3 Skill体系：Agent的"可插拔能力"

Skill是这个平台最底层的能力单元。每个Skill本质上是一个遵循Anthropic SKILL.md标准的指令包。

一个Skill的目录结构是这样的：

skills/
├── skill.yaml       # 元数据：名称、版本、作者
├── SKILL.md         # 核心指令：告诉Agent怎么做
├── tools/           # 工具函数（Python文件）
└── templates/       # 输出模板

啊对，看到 skill.yaml 和 SKILL.md 这个结构，你是不是觉得眼熟？对，就是Anthropic当初提出的Skill标准格式。我们直接沿用了这个标准，因为不用自创一套。

平台内置了四个Skill：流程分析、数据分析、风险识别、报告生成。每个Skill都有自己的SKILL.md，里面写清楚了这个Skill擅长什么、怎么用、输入输出是什么。当Agent被创建时，选中的Skill会被加载到system prompt里，成为Agent能力的一部分。

运行时加载Skill的代码如下：

for skill_name in selected_skills:
    skill = loaded_skills[skill_name]
    # 把SKILL.md内容塞进system prompt
    prompts.append(
        f"\n### Skill: {skill.name}\n{skill.instructions}"
    )

full_prompt = base_prompt + "\n".join(prompts)

说白了就是这么回事——把Skill的指令文档拼到系统提示词里，Agent就知道了这个能力该怎么用。简单粗暴但有效。

3.4 其他核心能力

四、一个PM为什么要学AI Coding？

好，产品介绍完了，聊点私货。

我本身是AI产品经理，不是工程师出身。以前我做产品就是画原型、写PRD、跟研发battle。代码？我不碰的。但2025年开始，AI Coding工具（Cursor、Claude、Copilot这些）的成熟度实在是到了一个拐点，我就想——要不我试试自己写？

但学AI Coding不能只学理论，你得有个够复杂的项目来练手。就像学游泳不能只在岸上比划，得下水。那下水就得选个水深一点的地方——所以我选了造一个端到端的AI智能体平台。

为什么说它复杂？你看啊：

  • 前端要写React交互，有路由、状态管理、Canvas分栏拖拽、SSE流式渲染
  • 后端是NestJS + TypeORM + SQLite，有完整的CRUD和AI编排
  • Python运行时要用FastAPI + Deep Agent SDK + LangGraph
  • 要跟百炼API对接，处理流式和非流式请求
  • 前中后三层之间要做SSE通信
  • 还要考虑Docker化部署、Cloudflare Pages、Railway

你看，前中后三层，三种语言（TypeScript、Python、JavaScript），三种框架（NestJS、FastAPI、React），还得对接大模型API。一个项目把前端、后端、AI、运维全串起来了。

五、对产品经理来说，AI Coding到底值不值？

这个话题可能是很多PM同行关心的。我说几个真实感受：

第一，AI Coding不是让你变成工程师，而是让你不再"被动"。以前我跟研发说"这个功能加一下"，对方说"做不了"，我就信了。现在我自己写一遍就知道到底做得了做不了，属于哪种做不了——是技术真不行、还是不想做、还是时间不够。这种判断力对PM来说价值巨大。

第二，你能把产品的"最后一公里"走完。以前我做PM，产品交付到"PRD评审通过"就结束了。现在写代码、部署、上线、甚至写公众号宣传，整个链路我都能参与。这种完整感是做PM很难体会到的。

第三，AI Coding的学习曲线确实陡，但没想象中那么可怕。尤其是有产品思维打底的情况下——你知道要做什么、为什么做、做成什么样算好。AI帮你解决的是"怎么写"的问题，而不需要AI帮你解决"是什么"的问题。后者才是PM的看家本领。

当然，也有代价。我大概花了半年时间，每天下班后和周末都在写。中间踩了无数坑——LangGraph的图编排理解错了，SSE的格式搞反了，百炼的API认证一直报错，Docker镜像反复build了十几版。这些东西如果是工程师来做可能一两天就搞定了，我自己折腾可能要一两周。

但反过来想——如果我不学AI Coding，这些东西可能永远都不会在我的能力范围之内。现在我至少知道了整个链路是怎么回事，下次跟团队讨论的时候，我可以说人话、也可以说技术话。
